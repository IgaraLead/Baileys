# Baileys Development Guidelines

Baileys is the **WhatsApp Web API client library** (TypeScript) used as a sidecar by Nexus for WhatsApp channel support. Fork of `@whiskeysockets/baileys` with IgaraLead customizations.

## Tech Stack

- **Language**: TypeScript 5.x (strict, `verbatimModuleSyntax`)
- **Runtime**: Node.js 20+
- **Core deps**: libsignal (E2E encryption), protobufjs, ws (WebSocket), pino (logging)
- **Testing**: Jest (ts-jest with ESM)
- **Formatting/Lint**: Prettier (tabs, 120 chars, no semi, single quotes) + ESLint flat config

## Build / Test / Lint

```bash
npm run build           # tsc + tsc-esm-fix
npm run lint            # tsc + eslint
npm run lint:fix        # prettier + eslint --fix
npm run test            # jest (unit tests)
npm run test:e2e        # jest --testMatch '**/*.test-e2e.ts'
npm run example         # tsx ./Example/example.ts
```

## Architecture

- **`src/Socket/`**: Core WebSocket client — `socket.ts`, `chats.ts`, `messages-send.ts`, `messages-recv.ts`, `groups.ts`, `business.ts`, `communities.ts`, `newsletter.ts`, `mex.ts`, `Client/`
- **`src/Utils/`**: Session management, encryption, parsing — includes `igara-session-manager.ts` (IgaraLead multi-tenant sessions)
- **`src/Types/`**: TypeScript type definitions (Auth, Chat, Contact, Events, Message, Socket, State, etc.)
- **`src/Signal/`**: Signal protocol implementation
- **`src/WABinary/`**: WhatsApp binary protocol encoding/decoding
- **`src/WAM/`**: WhatsApp metrics
- **`src/WAUSync/`**: WhatsApp user sync
- **`WAProto/`**: WhatsApp protobuf definitions (generated, don't edit manually)
- **`lib/`**: Compiled output (generated by build, not source)
- **Entry point**: `src/index.ts` — exports `makeWASocket` as the main client constructor

## Sidecar (`sidecar/`)

Express 5 HTTP wrapper exposing Baileys functionality to Nexus:

- **Entry point**: `sidecar/src/index.ts` — Express app (port 3500)
- **Files**: `session-manager.ts`, `message-utils.ts`, `logger.ts`
- **Endpoints**:
  - `GET /health` — health check (unauthenticated)
  - `POST /sessions/start` — start WhatsApp session (QR code flow)
  - `GET /sessions/:id/status` — session status
  - `POST /sessions/disconnect` — disconnect session
  - `POST /messages/send` — send WhatsApp message (text, media, quoted)
  - `GET /media/*` — serve downloaded media files
- **Security**: `X-Api-Key` auth (timing-safe comparison), input validation (regex), rate limiting (60 RPM sessions, 300 RPM messages)
- **Dev**: `npm run dev` (tsx watch) inside `sidecar/`

## Conventions

- **IgaraLead customization**: `tagSession()` in `src/Utils/igara-session-manager.ts` injects `client_slug` for multi-tenant session segregation
- **No external REST API** — sidecar is internal only (Nexus communicates via REST + webhooks)
- Keep upstream compatibility — isolate custom code to avoid merge conflicts
- Sessions stored at `/sessions/{slug}/{id}/` for per-client isolation

## Ecosystem Integration

- **Nexus**: Baileys runs as a sidecar process; Nexus sends/receives via `BAILEYS_SIDECAR_URL`
- **Webhooks to Nexus**: `message`, `status`, `qr`, `connection`, `contact`, `group` events → `NEXUS_WEBHOOK_URL`
- **Auth**: `BAILEYS_SIDECAR_API_KEY` protects the sidecar API
- **Session restore**: On startup, sidecar auto-restores previously authenticated sessions

## Post-Change Verification (MANDATORY)

After ANY code modification, run the full verification pipeline before considering the task done. Do not skip steps.

### Library Verification

```bash
npm run build                                    # TypeScript compile
npm run lint                                     # tsc + eslint
npm run test                                     # Jest unit tests
```

### Sidecar Verification

```bash
cd sidecar
npm test                                         # node:test runner (security + functional)
npx tsc --noEmit                                 # Typecheck
```

### Security Test Coverage (`sidecar/src/__tests__/sidecar.test.ts`)

The sidecar security test suite covers:
- **Authentication**: API key requirement on all protected endpoints, timing-safe key comparison, wrong/empty key rejection
- **Session ID validation**: empty, missing, special characters, path traversal, length limits
- **JID validation**: format enforcement, SQL injection in JID, domain suffix requirement
- **Message validation**: type enforcement (object required), null rejection
- **Health endpoint**: public access, no sensitive config leakage, JSON content type
- **Response security**: all error responses are JSON, no stack trace leakage
- **Media path security**: path traversal rejection, null byte injection prevention

### Security Principles

- `X-Api-Key` required on all endpoints except `/health`
- Timing-safe comparison (`crypto.timingSafeEqual`) for API key validation
- Input validation via regex on session IDs and JIDs
- Rate limiting: 60 RPM sessions, 300 RPM messages
- Request body limit: 10 MB
- No sensitive config (API keys, webhook URLs) in health responses
- Sessions isolated per `client_slug` at `/sessions/{slug}/{id}/`
- Media file serving must prevent path traversal
